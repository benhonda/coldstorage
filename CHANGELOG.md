# Changelog

## 2026-07-14

- feat(daemon): **the upload engine writes nothing to disk — it encrypts straight into the multipart upload.** Staging is gone (`UploadEngine.init` drops `stagingDir`, `UserSession` swaps `staging/` → `scratch/`): ciphertext now flows source → 4 MiB frame → 64 MiB part → S3 via a new `PartShipper` actor holding only the part in flight, so backing up a 40 GB video needs 40 GB of *upload*, not 40 GB of free space. `EnvelopeCipher.encryptedSize(ofPlaintext:)` supplies the progress-bar denominator staging used to measure.
- fix(daemon): **`AsyncThrowingStream { cont in … yield }` was buffering whole files into RAM** (unbounded policy + synchronous producer; a 256 MiB file → 391 MiB RSS — it killed a 1k-file deposit). New `ByteStreams.swift` holds the two sanctioned shapes — `pullStream(of:)` for a file, `scratchFileStream(at:write:)` for a PUSH source (PhotoKit, drained to the per-user scratch dir + swept at session build). New `StreamBackpressureTests` pins RSS to a number; `daemon:test` now runs `--no-parallel` so Argon2id's memory-hard KDF can't pollute the measurement.
- fix(daemon): **a source that changed since the scan is rejected, not archived** — `archive` re-checks the streamed plaintext SHA-256 against a new `IngestItem.expectedSha256` (`nil` for a Photos asset) and fails the blob with `ColdStorageError.contentDrift`; without it a file edited mid-upload passed every downstream check (`verify` is only a HEAD) and surfaced as corruption at restore.

- fix(security): **cross-account leak — the daemon's journal was machine-wide; state is now per-user (`UserSession`).** The journal was a single `~/Library/Application Support/ColdStorage/coldstore.sqlite` with no owner column, and sign-out dropped only the AWS credentials + the MasterKey — so signing out of account A and into account B on the same Mac showed B **all of A's files, folder paths, sizes and watched-folder registry**, which the run loop then kept scanning under B's session (new files in them would have been archived into B's vault). File **bytes** never crossed accounts (IAM scopes S3 to `blobs/<identity>/*`, MasterKeys are escrowed per user-pool `sub`, Deep Archive thaw is backend-gated) — the *index* did. The fix is structural, not a patch: new `UserSession` (`ColdStorageCore/UserSession.swift`) owns **all** per-user state (journal, staging dir, `status.json`, the `SwappableKeyProvider` MasterKey holder, the `VaultPrefix`, both engines), built at `authenticate` and destroyed at `deauthenticate`; `DaemonService` now holds one `private var session: UserSession?` and **nothing unscoped**, so signed out the reads answer empty and every mutation throws *"not signed in"* — there is no unscoped journal for a future subsystem to forget to scope. On disk, state moves to `<dataRoot>/users/<sub>/{coldstore.sqlite, staging/, status.json}`, keyed by the Cognito **user-pool `sub`** (the canonical identity; `identityId` is a derived S3-addressing detail). `getStatus` gains `signedIn: boolean`. **UI:** the renderer now resets every vault-derived slice (files/status/excludes/run/failures/restores) on sign-out **and** on an account switch (`reducer.ts` `authChanged`, keyed on the account — the daemon's isolation is only half the fix if the renderer still holds the last user's tree). New `SessionIsolationTests` drives the real `DaemonService` (sign in as A → deposit → sign out → sign in as B → B sees nothing; signed-out reads empty / writes refused; watched-folder + exclude isolation) + new account-switch tests in `reducer.test.ts`. **Run `task daemon:mac:reset:local` once:** it now wipes `<dataDir>/users/` *plus* the orphaned pre-refactor machine-wide files.
- fix(security): **the storage-quota gate had been silently inert since it shipped** — same root cause, one layer over. `S3Store.usageBytes` was passed a bare `blobs/<identityId>` prefix, but the IAM `s3:ListBucket` grant is conditioned on `s3:prefix` matching `blobs/<sub>/*` — the un-slashed form doesn't match, so every usage listing came back `AccessDenied`, `bytesStored` was always `null` in production, and `hasCapacity` failed open every time. New typed `VaultPrefix` (`VaultPrefix.swift`) kills the bug by construction: `.key(for: blobId)` (no trailing slash) vs `.listing` (**with** it — what `ListObjectsV2` and the IAM condition need). `BlobPlan.keyPrefix: String` → `BlobPlan.prefix: VaultPrefix`, `UploadEngine.run(keyPrefix:)` → `run(prefix:)`, `S3Store.usageBytes(prefix: VaultPrefix)`; `bytesStored` is non-null whenever signed in. New slash tests in `PerUserPrefixTests`.
- refactor(daemon): **no more silent dogfood fallback — `coldstored` states its identity or refuses to start.** It now requires **exactly one** of: Cognito configured (multi-user), or an explicit `COLDSTORE_DEV_IDENTITY=<name>` (local dev/MinIO); with neither it prints a refusal and `exit(2)`. Previously, missing Cognito config silently fell back to the default AWS credential chain — the dogfood IAM user, whose policy grants `blobs/*`, i.e. **every** user's objects — plus a flat `blobs` prefix (AVOID4: a fallback that quietly widens a security boundary is not a convenience). The daemon now takes a data **root**, `COLDSTORE_DATA_DIR`, replacing `COLDSTORE_JOURNAL`/`COLDSTORE_KEK`/`COLDSTORE_STAGING`/`COLDSTORE_STATUS` (`COLDSTORE_SOCKET` still stands alone — it's machine-level); the launchd plist template and the dev daemon env (`COLDSTORE_DEV_IDENTITY: local` + `COLDSTORE_DATA_DIR: .dev-data`) follow. Docs synced: `coldstorage/DESIGN.md` (session model, per-user on-disk layout, `VaultPrefix`), `coldstorage/README.md` (identity + env), `ui/DESIGN.md` (daemon contract: `signedIn`, signed-out-serves-nothing, session lifecycle), root `README.md`, `PROD.md`.
- refactor(ui): **Settings shows ONE storage number, not two.** "In deep storage" summed the local file rows (a *per-device* figure) while "Plan usage" listed S3 (a *per-identity* figure) — two numbers that can honestly disagree for a multi-device user, sitting side by side, each labelled as the truth about the vault. They now collapse into one row, *"In deep storage — 12 GB of 25 GB"*, sourced from `getStatus.bytesStored` (the live S3 listing the quota is actually enforced against), so the Storage card and the downgrade warning can never disagree. This pair disagreeing is how the cross-account leak above first showed itself: the journal sum was inflated with a second account's files while the S3 total was not. Selection sizes (request-a-copy, My Files) still sum file rows — a different question.
- refactor(daemon): **`RestoreEngine` depends on a protocol, not the concrete `S3Store`.** `UploadEngine` had a `BlobStore` seam from day one; restore didn't — so anything constructing a `UserSession` had to stand up a real AWS client (and its CRT TLS context, which under load can fail to initialize outright and made the new isolation suite flaky). New `VaultStore` protocol (`thawState`/`requestThaw`/`getRange`/`usageBytes`) + `Vault = BlobStore & VaultStore`; `S3Store` conforms to both; `SessionIsolationTests` now runs against a fake vault with no AWS at all.
- chore: `backend:db:reset` + `scripts/db-reset.ts` removed — under zero-knowledge, wiping the accounts table destroys the only copy of each user's wrapped MasterKey, so a "clean slate" task is a foot-cannon with no safe use.
- feat(build): **`app:mac:run:*` / `app:mac:package:*`** — one command runs the whole app against a named lane (staging / local / production, never a default), reinstalling the daemon first so a stale launchd binary can't fake a broken feature. Gated by `app:mac:_require-api-deployed`: refuses when `account-backend/` differs from what the lane's branch deployed (the app calls the DEPLOYED API, never your local one).
