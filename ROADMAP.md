## What it is
A Mac app that works like a **remote SSD for the stuff you can't lose**: drag photos/files in, they're encrypted on-device and archived to S3 Glacier Deep Archive; get them back later with a quoted wait + fee. Convenience layer on S3 for the non-technical "I can't do AWS" user.

## Decisions in force (don't re-litigate)
- **Build to dogfood, skip demand validation** тАФ Ben is target user #1. Build the real V1, defer the commercial layer.
- **Stack:** Electron/React UI (control panel) + a **standalone Swift upload daemon** (launchd). NOT react-native-macos, NOT Tauri/Rust.
- **UI is a thin client over the control socket** тАФ Electron's main process speaks the JSONL protocol directly (Node `net.Socket`), no `coldstorectl` spawn / native bridge. The control protocol IS the UI contract. See [ELECTRON-UI-DESIGN.md](./ELECTRON-UI-DESIGN.md).
- **Upload robustness is the V1 crown jewel** тАФ built bulletproof, not deferred.
- **Correctness before speed** тАФ get it right + proven, *optimize later*. Cross-blob concurrency/throughput is explicitly deferred until there's a real-AWS bench to measure against (local MinIO can't show the win).
- **Brand:** calm, plainspoken, quietly warm, straight. No catastrophe imagery; no privacy over-claims beyond V1; "remote SSD" is a metaphor, never a literal claim.

## Built & PROVEN тЬЕ (verified end-to-end against MinIO)
The Swift package in [`coldstorage/`](./coldstorage/):
- **Pipeline:** scan тЖТ per-file SHA-256 тЖТ AES-GCM frame encryption (per-blob DEK, deterministic counter nonces) тЖТ batch small files into locality blobs тЖТ **resumable S3 multipart** тЖТ HeadObject verify тЖТ **SQLite/WAL journal** (the SPOF).
- **Restore:** thaw-aware + idempotent тАФ Deep Archive `RestoreObject` thaw (when needed) тЖТ ranged GET тЖТ decrypt тЖТ reassemble тЖТ hash-verify (`coldstore-restore`; re-run until it lands). **PROVEN END-TO-END on the real prod vault тЬЕ (2026-06-27):** the first live Deep Archive thaw completed the full clock and `restore` returned `state=restored` with a written file тАФ and because `RestoreEngine` won't write on a hash mismatch, that written file *is* the byte-identical proof. The whole pipeline (depositтЖТencryptтЖТmultipartтЖТDEEP_ARCHIVEтЖТthawтЖТranged GETтЖТdecryptтЖТverify) now has zero unproven legs against real AWS.
- **Verified by real checks:** unit tests pass ┬╖ encryption is genuine ciphertext ┬╖ **resume across a hard kill** (same uploadId reused, zero orphans) ┬╖ idempotent re-runs ┬╖ **round-trip byte-identical** for solo *and* batched/offset files.
- **Control plane (verified end-to-end vs MinIO):** registry-driven `coldstored` actor + portable unix-socket JSONL IPC (`ControlServer`/`ControlClient`, socket `0600` = owner-only). Commands `getStatus ┬╖ listSources ┬╖ listFiles ┬╖ addSource ┬╖ removeSource ┬╖ previewDeposit ┬╖ deposit ┬╖ depositPhotos ┬╖ movePath ┬╖ deletePath ┬╖ createFolder ┬╖ listExcludes ┬╖ addExclude ┬╖ removeExclude ┬╖ getPricing ┬╖ triggerNow ┬╖ restore ┬╖ pauseSource ┬╖ resumeSource ┬╖ ping`; pushed events `runStarted ┬╖ fileArchived{file,blob} ┬╖ uploadProgress{file,path,bytes,totalBytes} ┬╖ runFinished{тАж,blobsFailed} ┬╖ blobFailed{blob,kind,message,paths} ┬╖ sourcesChanged ┬╖ filesChanged{moved,to|deleted|created} ┬╖ restoreRequested/restoreInProgress/restoreCompleted ┬╖ paused/resumed ┬╖ error`. **Reorganize/delete over IPC (verified vs MinIO):** `movePath from=<vault/path> to=<vault/path>` is the single primitive behind file/folder **move AND rename** (a rename is a move to a sibling path) тАФ a cheap journal `relativePath` prefix-sweep (no S3, no thaw, the blob never moves; the stable `id` тАФ the upsert dedup key тАФ is preserved, so a rescan won't re-upload); `deletePath path=<vault/path>` **tombstones** the subtree (`status=deleted`, row + blob mapping kept for a deferred repack/GC, dropped from `listFiles`). Both emit `filesChanged` тЖТ the UI re-reads `listFiles`. Proven live: renameтЖТmove-folderтЖТdelete-fileтЖТdelete-folder all reflected in `listFiles`, with the raw journal confirming id-stability + tombstone-keeps-row (`task daemon:move-ipc FROM= TO=` / `daemon:delete-ipc PATH=`). **Restore over IPC (verified vs MinIO):** `restore file=<id> out=<path> [tier days]` drives the idempotent `RestoreEngine` over the socket тАФ one step per call (request thaw тЖТ report progress тЖТ download+verify), pushing a progress event for a live watcher; proven byte-identical over the socket (`task daemon:restore-ipc FILE=<id>`). Sources are a **journal-backed registry (SSOT)** тАФ `COLDSTORE_SOURCES` is just a seed; add/remove survive restart (proven: relaunch w/ no env тЖТ folders persist, no re-upload). **launchd LaunchAgent** plist template + `task daemon:install`/`daemon:uninstall` (RunAtLoad+KeepAlive). Drive it with `coldstorectl` (`task daemon:run` / `daemon:ctl -- getStatus`; `coldstorectl <sock> watch` for live events).
- **Graceful error handling (verified vs MinIO):** transient retry is the **AWS SDK's** job (built-in backoff тАФ we don't re-implement it). Our layer **classifies** every failure (`FailureKind`: permanent vs transient тАФ SSOT in `Failure.swift`) and **isolates per-blob** тАФ a poison blob is recorded + surfaced as a `blobFailed` event and the run *continues*; permanent failures (the `InvalidStorageClass`/`NoSuchBucket` class) go on an in-memory skip list so the daemon stops re-staging a doomed blob, and show in `getStatus.permanentlyFailedBlobs`. The engine takes `any BlobStore` (DI seam). **Proven:** unit tests (classification + real-pipeline fault-injection via a fake `BlobStore`) + live тАФ daemon survives a real `NoSuchBucket`, pushes `blobFailed{kind:permanent}`, keeps answering, archives nothing falsely. *(In-memory skip only тАФ persisting it needs a journal schema change, deferred.)*
- **Dev loop (no Docker):** native Swift (swiftly) + MinIO binary. `task daemon:setup тЖТ daemon:minio тЖТ daemon:build:dev тЖТ daemon:test тЖТ daemon:archive`. Live daemon: `task daemon:run` then `task daemon:ctl -- getStatus`.
- Spikes that de-risked it: [`phase0-upload-spike/`](./phase0-upload-spike/), [`phase0-photos-spike/`](./phase0-photos-spike/) (**RUN + PROVEN 2026-06-26** on a real Mac тАФ durable Photos TCC grant under launchd + full-res original incl. iCloud download).

## Dev environment & gotchas (read before building тАФ saves hours)
- **No Docker here.** Toolchain is **native**: Swift via `swiftly` (installed to `~/.local/share/swiftly`), MinIO + `mc` as plain binaries. `task daemon:setup` installs both idempotently; `.devcontainer/post-create.sh` does it on rebuild.
- **`swift` on PATH:** new shells source `~/.local/share/swiftly/env.sh` (wired into `~/.zshrc`/`~/.bashrc`). If `swift: command not found`, `source` that file or open a new terminal.
- **SwiftPM build lock тАФ THE footgun:** if a `swift build`/`swift test` is killed mid-flight, a leftover process can hold `.build/.lock` (a dotfile) and **every later build blocks at 0% CPU, looking "hung."** Now self-healing: `task daemon:build:dev`/`daemon:test` depend on `daemon:unlock` (clears a stale lock when no build/test is live). Manual fix: `task daemon:unlock` (or `pkill -9 -f swift-build; pkill -9 -x swift-test; rm -f .build/.lock`). **Don't wrap `swift test`/`swift build` in `timeout`** тАФ the swiftly `swift` shim detaches `swift-test`, so `timeout` reaps the shim while the real worker (and its `xctest` child) runs on unbounded. Reap by name instead (`pkill тАж swift-test`/`ColdStoragePackageTests`).
- **Flaky test hang тАФ FIXED 2026-06-20 (was swift-corelibs-XCTest, not our code).** Symptom: I/O-touching tests hung at 0% CPU (~15% idle, ~100% under load). gdb backtrace pinned it to the main thread stuck in `XCTest.awaitUsingExpectation тЖТ CFRunLoopRun тЖТ ppoll` тАФ a known unresolved XCTest-on-Linux deadlock ([swift-corelibs-xctest#504](https://github.com/swiftlang/swift-corelibs-xctest/issues/504)): the CFRunLoop driving each test's completion occasionally loses its libdispatch wakeup. **Fix: migrated the whole suite to swift-testing** (`@Suite`/`@Test`/`#expect`/`#require`) тАФ Swift-Concurrency-based, no CFRunLoop. Stress-verified 20/20 clean (0.02s/run). `task daemon:test` is back to a plain `swift test`. Don't reintroduce XCTest (`import XCTest`/`XCTestCase`) тАФ it brings the deadlock back.
- **Debugging wedged Swift procs:** use **gdb** (installed + in post-create): `sudo gdb -p <pid> -batch -ex 'thread apply all bt'`. The toolchain's lldb is broken here (wants libpython3.12; box has 3.14).
- **Builds are ~60s** (debug-linking the full AWS SDK); *incrementals are seconds* once warm. Don't spawn parallel builds тАФ they contend.
- **Inspect the journal:** `sqlite3 coldstorage/coldstore.sqlite` (install `sqlite3` CLI via apt; the daemon itself uses libsqlite3 directly).
- **MinIO console:** http://localhost:9001 (`minioadmin`/`minioadmin`). Storage class is conditional тАФ `DEEP_ARCHIVE` on real AWS, omitted for MinIO (which rejects it).
- **VS Code port-forward hijacks native MinIO on the Mac (cost ~15 min, 2026-06-26):** the devcontainer once ran MinIO on 9000/9001, so VS Code *remembered* those forwards and resurrects them on every reattach тАФ a host listener on `127.0.0.1:9000` with nothing behind it. Native MinIO then can't bind 9000 тЖТ `mc alias set` hangs forever, looking like a wedged task. Fix shipped: `.vscode/settings.json` тЖТ `"remote.restoreForwardedPorts": false` (**but `.vscode/` is gitignored тАФ this is local-only, won't travel via git; a fresh devcontainer needs it re-applied**). Symptom check: `lsof -nP -iTCP:9000 -sTCP:LISTEN` showing `Code Helper` = the hijack; remove the forward in the Ports panel.
- **Cold-build `swift run` contention (daemon + ctl):** `task daemon:ctl`/`daemon:fsevents-test` etc. each invoke `swift run <product>`, which acquires the SwiftPM `.build` lock to build first. On a *fresh* tree the daemon's `coldstored` build (~80s) and a concurrent `coldstorectl` build (~80s) serialize on that lock тАФ the second prints "Another instance of SwiftPM (PID тАж) is already running, waitingтАж" and looks hung. It isn't; it proceeds once the first finishes. To avoid the wait, run `task daemon:build:dev` once up-front (builds all products), *then* start the daemon + ctl.
