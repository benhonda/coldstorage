# i18n — inline en/fr, no locale files

Bilingual (English + French) content written **inline at the call site**, not in JSON
locale files.

**Read when:** adding translatable copy, the language switcher, or the `/fr` routing.

## Contract
- Two languages chosen at the call site by `t(en, fr)`. No locale files, no string keys.
- URL rule: English has **no prefix** (`/`, `/jobs`); French is under **`/fr`**. A
  `($lang)` route segment carries the language.
- Server: `langUtils(params).t(en, fr)`. Client: `useI18n().t(en, fr)` for strings,
  `tc(bilingual)` for JSX.

## Non-negotiables
| key | rule | why |
| --- | --- | --- |
| url-rule | English unprefixed, French at `/fr`; absent = English; no `/en` ever. The router's `($lang)` is a **generic optional segment** — fr-only is enforced in `langUtils`, and static routes out-rank it | the fixed routing convention (build paths via the router — see `references/routing.md`) |
| t-identity | `t<T>(en: T, fr?: T): T` returns one arg by active language; fr falls back to en | same shape on server (`langUtils`) and client (`useI18n`) |
| bilingual-jsx | non-string content uses `tc` + `BilingualContent<T> { en: T; fr?: T }` | JSX/ReactNode translations stay typed |
| no-locale-files | never extract strings into JSON/resource files | inline is local (both languages at the decision point), type-safe (missing fr = type error), and typo-proof — the whole point of the convention |
| switch-preserves-url | language switch rewrites only the prefix, keeping path/search/hash | predictable navigation |

## Shape — write fresh (illustration, not gospel)
```ts
// server (loader/action)
const { lang, t, isFrench } = langUtils(params);
const title = t("Welcome", "Bienvenue");
// client
const { t, tc, switchLanguage, isFrench } = useI18n();
<h1>{t("Hello", "Bonjour")}</h1>
<p>{tc({ en: <Strong>Hi</Strong>, fr: <Strong>Salut</Strong> })}</p>
```
